# 写作规则与约束

## 一、基本原则

1. 每次开始写作前，必须先理解 `ORIGINAL_BRIEF.md` 中的原始写作委托。
2. 先确认大纲，再写正文。
3. 每章工作底稿单独一个 MD 文件。
4. 每章正式正文单独一个 MD 文件。
5. 每章先写章节大纲，确认后再写正文。
6. 正文必须服务观点、证据、数据、事实，不写空泛口号。
7. 全书使用倒叙：先写事实和变化，再解释原因和体系。

## 一点五、成果优先原则

全书必须把“需求到交付成果”作为核心变化来写，而不是“需求到系统”。

1. AI 时代，软件、页面、代码和流程配置会越来越容易生成。
2. 但用户成果仍然很难，需要业务理解、数据治理、流程嵌入、组织采用、持续运营和结果验证。
3. 系统只是手段，成果才是终点。
4. FDE 的价值不是更快做出软件，而是更可靠地把需求推进到用户可感知、可运行、可验证的业务成果。
5. 后续章节遇到“系统交付”“系统搭建”“可运行系统”等表述时，优先改成“成果交付”“交付闭环”“业务结果”“可验证成果”等表达。

## 二、文风要求

目标文风：

有技术实践经验、有宏观行业视野、有行业分析能力的创业者叙述。

要求：

1. 语言要克制、准确、具体。
2. 避免“AI 是未来”“赋能千行百业”这类套路表达。
3. 避免堆概念、堆形容词、堆愿景。
4. 每一节都要有清晰判断。
5. 每个判断后面尽量跟事实、数据、案例或机制解释。
6. 章节之间要有自然过渡，不能像文章拼接。

## 三、内容边界

必须基于现有材料整理和扩展：

1. `FDE_大纲-上.MD`
2. `FDE_大纲-下.MD`
3. `运维实践.txt`
4. `运维实践 - 总结.txt`
5. `D:\ai-source-code` 中团队实践相关项目和文档

可以结合外部行业数据，但必须标明来源和年份。

不能凭空编造：

1. 客户数量。
2. 收入金额。
3. 项目金额。
4. 人员规模。
5. AI 效率提升倍数。
6. 客户评价。

如果需要这些信息，先标为“待作者确认”。

## 四、平台与产品表述规则

不要把“云镜低代码平台”写成对外推销的软件产品。

推荐表述：

1. 团队资源数字化平台。
2. 数据治理工具。
3. 交付基础设施。
4. AI-FDE 的内部操作系统。
5. 团队用于沉淀业务对象、流程、权限、数据关系和交付资产的基础设施。

核心判断：

过去，它可能更像一个软件产品；在 AI 时代，它退回到了基础设施位置。

它本身不再是最值钱的前台商品，但非常重要。客户真正购买的不是平台本身，而是更快、更深、更可持续的 AI 落地能力。

## 五、案例处理规则

案例必须加工润色，不能原样摊开内部项目材料。

要求：

1. 先抽象机制，再呈现事实。
2. 不写成产品说明书。
3. 不写成项目履历清单。
4. 不暴露敏感客户、服务器、账号、内部凭证、真实访问地址。
5. 项目名称可以视情况泛化，例如“某复杂实验室排程项目”“某多环境生产运维体系”。
6. 案例最终必须落到行业启示。

案例写作结构：

1. 当时的问题是什么。
2. 传统方式会怎么做。
3. FDE 体系如何处理。
4. 用到了哪些基础设施和工具。
5. 产生了什么变化。
6. 对行业有什么启示。

## 六、数据使用规则

外部数据必须满足：

1. 来源可靠。
2. 年份明确。
3. 只用于支撑判断，不堆砌。
4. 不过度引用长段原文。
5. 在章节末尾保留来源列表。

优先数据源：

1. Gartner。
2. McKinsey。
3. Stanford AI Index。
4. IDC。
5. 其他行业报告需单独判断可信度。

## 七、文件分层规则

从现在开始，工作底稿和正式正文必须分离，不能混写在同一个文件中。

目录分工：

1. `book-outline/chapters/`：只放章节工作底稿，包括章节定位、核心问题、核心观点、小节结构、可用素材、事实底稿、待确认问题、上下章过渡。
2. `book-outline/manuscript/`：只放干净正文，包括前言、各章正文和结语。

硬性要求：

1. 正文必须单独成文，不能放在章节底稿的“正文草稿区”里。
2. 正文文件中不保留事实底稿、内部备注、待确认问题、工作流说明和未整理素材。
3. 章节底稿可以链接对应正文文件，但不能混入正文全文。
4. 后续写作时，先在 `chapters` 中整理事实和结构，再在 `manuscript` 中写干净正文。
5. 任何章节进入正式写作后，都要同步更新 `WRITING_STATE.md`，注明对应的正文文件路径。

## 八、章节底稿文件规则

章节底稿文件命名：

`CH##_章节标题.md`

示例：

1. `CH01_当前事实_产业AI已经进入落地分水岭.md`
2. `CH09_团队资源数字化平台_AI-FDE背后的基础设施.md`

每章底稿文件必须包含：

1. 章节定位。
2. 核心问题。
3. 核心观点。
4. 小节结构。
5. 可用素材。
6. 需要补充确认的问题。
7. 与上一章和下一章的过渡。
8. 对应正文文件路径。

正文文件命名：

`CH##_章节标题.md`

正文文件必须是干净成稿，只保留：

1. 章节标题。
2. 正文内容。
3. 必要的小节标题。
4. 本章资料来源。

## 九、中断与恢复规则

任何一次中断后，继续工作前先读取：

1. `README.md`
2. `ORIGINAL_BRIEF.md`
3. `WRITING_CONTEXT.md`
4. `WRITING_RULES.md`
5. `WRITING_STATE.md`
6. 当前正在处理的章节文件

恢复后先做一件事：

在 `WRITING_STATE.md` 中确认当前任务、已完成内容和下一步。

## 十、图文排版与出版图版规则

图文 HTML、PDF、公众号长文等对外阅读版本，默认采用“严肃技术实践书 / 出版图版”风格。

核心判断：

1. 本书不是 SaaS 产品介绍页，也不是低代码平台宣传页。
2. 图示的任务是解释结构、流程、架构和判断，不是制造装饰感。
3. 视觉语言要服务“产业现场、技术实践、交付成果、组织方法论”的叙事气质。

必须避免：

1. 大面积圆角卡片。
2. 卡片套卡片。
3. 强阴影和悬浮模块。
4. 泛 AI 高饱和蓝紫渐变。
5. 产品后台式模块堆叠。
6. 过度装饰的光效、色块和氛围背景。
7. 让图示看起来像软件功能介绍，而不是书内解释性图版。

推荐做法：

1. 使用白底或近白底阅读面。
2. 用细规则线组织图表层级。
3. 使用深灰蓝、近黑和克制墨绿色作为主结构色。
4. 流程图、闭环图、架构图优先采用图版行结构、分层结构、规则线矩阵或出版物式表格。
5. 图前楔子用规则线和轻量左侧强调处理，不做圆角提示卡。
6. 图题、图体、图注要形成清晰的出版物层级。
7. 移动端优先保证纵向阅读，不强行保留桌面端卡片布局。
8. 每次重排后必须抽查移动端 HTML 截图和 PDF 图表页。
