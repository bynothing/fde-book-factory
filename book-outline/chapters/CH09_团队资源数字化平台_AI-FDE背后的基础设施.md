# CH09 团队资源数字化平台：AI-FDE 背后的基础设施

## 章节定位

下篇开场章。重估团队资源数字化平台在 AI 时代的位置：不是前台售卖产品，而是 FDE 的内部基础设施。

## 核心问题

我们团队为什么能用小团队支撑复杂产业项目？

## 核心观点

1. FDE 需要的不是一个拿去售卖的软件产品，而是一套可快速调用、可持续沉淀的内部基础设施。
2. 这套基础设施在传统软件时代可能被包装成产品；到了 AI 时代，更准确的定位是团队资源数字化平台、数据治理工具和交付操作系统。
3. 它不再是最值钱的前台商品，但仍然极其重要。
4. MCP 工具链让这些基础设施能力可以被人和 Agent 调用。

## 小节结构

1. 从软件产品到内部基础设施的价值重估。
2. 数据结构治理：业务对象、字段、关系、状态和流程数据。
3. BPM、菜单、权限、租户：企业系统治理能力。
4. MCP：把基础设施变成可调用工具。
5. Web2 定制前端：承载复杂交互和场景体验。
6. 多客户体系与场景复用。
7. 价值重估的最终判断。

## 事实底稿

### 本章事实判断

1. CH09 是下篇开场章。从"原理"转入"落地"，第一站是基础设施。
2. 团队资源数字化平台在 AI 时代的价值位置发生了重估：它不再是前台售卖的产品，而是 FDE 的内部基础设施。
3. 价值重估不是贬低它，而是更准确地描述它在 AI-FDE 体系里的位置：它是底座，不是门面。
4. 这种位置变化对企业有现实含义：客户购买的不是一个软件产品，而是更快、更深、更可持续的 AI 落地能力。
5. 数据结构治理是基础设施的核心入口：业务对象、字段、关系、状态、流程数据，是 FDE 现场最频繁调用的能力。
6. BPM、菜单、权限、租户是基础设施的"四件套"，构成企业级系统治理能力。
7. MCP 工具链把上述能力封装为可调用的 CLI 和脚本，让 FDE 不必从零点界面，而是从命令行直接拉起。这是基础设施从"人用"到"人和 Agent 都能用"的关键。
8. Web2 框架承担低代码前端无法支撑的复杂交互和行业场景体验。
9. 多客户体系（多租户 + 模板租户 + 扩展模块）是 AI-FDE 体系跨项目、跨行业、跨客户复用的基础。
10. 没有这套基础设施，FDE 模式不可能规模化；有了这套基础设施，FDE 才能把时间花在判断、决策、价值提炼和客户对齐上。

### 原始大纲可用观点

1. FDE 需要的不是一个拿去售卖的软件产品，而是一套可快速调用、可持续沉淀的内部基础设施。
2. 这套基础设施在传统软件时代可能被包装成产品；到了 AI 时代，更准确的定位是团队资源数字化平台、数据治理工具和交付操作系统。
3. 平台价值从"产品功能"转向"数据治理、流程沉淀、工具调用、资产复用"。
4. 客户真正购买的不是平台本身，而是更快、更深、更可持续的 AI 落地能力。
5. 团队真正积累的不是一个个项目页面，而是一套可复用的行业交付能力。
6. 团队资源数字化平台 + MCP 工具链 + 数字员工运维 + 组件库 = FDE 工作所需的技术底座。
7. 数据结构治理是低代码系统构建工作流的起点，决定后续所有动作的拓扑和成本。
8. 多租户 + 模板租户 + 扩展模块 = 多客户复用的工程基础。

### 团队实践可用素材

1. 数据结构治理：业务对象、字段、关系、状态、流程、菜单、权限的统一入口。
2. 表单创建：通过 MCP 工具链，按数据关系矩阵和依赖顺序串行创建表单：先主数据，再被引用的基础对象，再业务单据，最后台账、统计、复盘类后置表单。
3. 字段设计：业务含义、数据类型、组件类型、是否固定枚举、是否引用业务对象、是否参与流程或统计。
4. 数据关系矩阵：来源表单、字段 ID、目标表单、关系、显示字段、带出字段、是否只读。
5. 依赖排序：被依赖对象先创建，目标表单先于来源表单；自引用、循环依赖或确实无法提前解析的关系进入后置补配清单。
6. 关联字段配置：associationForm 表达当前字段关联到哪个目标表单；followChange 表达当前字段跟随本表单中哪个字段变化。
7. 跟随联动：leaderFormId 必须是本表单字段 ID，不是目标表单 ID。
8. 表单审计：创建后立即检查实际平台保存结果是否与设计一致。失败示例：leaderFormId 错误地写成了目标表单 ID。
9. 菜单层级：业务菜单只允许两级，第一级模块目录，第二级具体功能；不创建业务根菜单或系统汇总菜单。
10. 菜单路径：表单菜单用 lowcode/dynamicInfo/{formId}，流程菜单用 lowcode/dynamicWorkflow/{modelId}。
11. 角色授权：菜单已创建但用户看不到的常见原因是只授权了子菜单未授权父菜单，或表单菜单授权了但流程菜单漏了。
12. BPM 流程：开始、用户任务、结束、网关；分配规则数量匹配用户任务数量。
13. 端到端业务测试：表单 CRUD、关联表单选择、跟随联动、菜单可见、图标显示、流程发起、审批、结束、数据清理。
14. 多租户数据隔离：MyBatis Plus TenantLineHandler 自动在所有查询中插入 WHERE tenant_id = ?；TenantContextHolder 通过 TTL 跨线程传递租户上下文。
15. 租户克隆：模板租户 → 新租户。saveTenantAsTemplate 序列化表单 + 流程 + 菜单 + 看板 + 系统配置，上传 OSS。createFromTemplate 从 OSS 拉取模板 JSON，创建空租户 → 复制角色 → 复制用户 → 拓扑排序解析表单依赖 → 逐个创建表单 → 复制流程模型 → 复制菜单、看板、系统配置。
16. 表单模板机制：表单定义序列化存储到 OSS，支持自动版本管理。
17. 低代码引擎：表单 JSON → 解析为字段定义 → 生成 DDL → 创建物理表。
18. 扩展模块：通过 extensions/ 目录 + Maven Profile 实现客户代码物理隔离。包名约定：cn.iocoder.yudao.module.{主模块}.extension.{扩展名}。
19. Web2 框架：多场景前端框架、低代码表单、Web2 定制前端。
20. 数字员工运维：基础设施的延伸，承担 7×24 巡检、备份、构建、部署、故障排查。

### 基础设施六层结构的内部关系

1. 数据结构治理层是入口：所有上层动作都基于稳定的数据结构。
2. 流程引擎层是血液：把数据节点、角色、状态、动作串成业务流。
3. 菜单权限租户层是骨架：让不同角色在不同租户看到不同入口。
4. MCP 工具链层是手脚：把上述能力封装为可调用工具，支撑 FDE 和 Agent 的调用。
5. Web2 框架层是皮肤：让复杂交互、行业场景、定制体验落地。
6. 多客户体系层是承重墙：让基础设施能跨项目、跨行业、跨客户复用。
7. 这六层不是平行的清单，而是嵌套的依赖：下层缺失，上层无法稳定；上层漂移，下层也会被连累。

### 价值重估的具体含义

1. 价值位置变化：从"前台售卖的产品"退到"内部基础设施"。
2. 价值评估方式变化：从"卖了多少 license"转向"支撑了多少项目、覆盖了多少客户、沉淀了多少资产"。
3. 价值主体变化：从"产品收入"转向"组织能力"。
4. 客户购买的变化：从"购买一个软件"转向"购买更快、更深、更可持续的 AI 落地能力"。
5. 团队积累的变化：从"积累一个个项目页面"转向"积累一套可复用的行业交付能力"。

### 数据结构治理的具体内涵

1. 业务对象建模：识别主数据、业务单据、明细数据、显示字段。
2. 数据关系矩阵：明确所有表单之间的引用关系和创建拓扑。
3. 字段详细设计：业务含义、数据来源、数据类型、组件类型、是否枚举、是否引用、是否参与流程或统计。
4. 依赖排序与创建：先主数据，再被引用基础对象，再业务单据，最后台账、统计、复盘类后置表单。
5. 关联字段与跟随联动补配：associationForm、followChange、leaderFormId 配置。
6. 表单审计与修复：检查组件类型、枚举选项、关联配置、跟随联动、只读展示字段。
7. 这一套流程在系统构建工作流里有标准步骤，FDE 在现场按这套流程做。
8. 数据结构治理的稳定，决定上层所有动作的稳定。

### BPM、菜单、权限、租户四件套的内部关系

1. BPM 流程：节点、分配规则、表单绑定、流程发布。
2. 菜单：模块目录、功能菜单、表单菜单、流程菜单，父子结构清晰。
3. 权限：角色定义、用户绑定、菜单授权、数据范围。
4. 租户：多租户隔离、模板租户、租户克隆、租户差异化管理。
5. 这四件不是孤立的组件，而是相互依赖的体系：流程绑定表单，菜单发布入口，权限决定谁能用，租户决定数据归属。

### MCP 工具链的具体内涵

1. 把数据结构、流程、菜单、权限、数据操作封装为可调用的 CLI 和脚本。
2. FDE 在终端里直接拉起批量操作，不必从零点界面。
3. Agent 在工作流中调用这些工具，把自动化和人工判断统一到同一个调用体系。
4. MCP 工具链让"人能调用"和"机器能调用"统一起来。
5. 没有 MCP 工具链，AI Agent 只能停留在对话窗口里，看不到业务闭环。

### Web2 框架的具体内涵

1. 多场景前端：行业工作台、复杂交互、GIS、可视化、看板。
2. 定制开发：在低代码前端无法支撑的体验上补充。
3. 与低代码的分工：能用低代码做的不要进入 Web2；Web2 专门做低代码做不了的事。
4. 项目级定制与平台级组件的分离：项目级定制进入 extensions/，平台级组件回到 web2 框架。

### 多客户体系与场景复用的具体内涵

1. 多租户：客户数据物理隔离，业务逻辑共享。
2. 模板租户：一个产品线对应一个模板租户，沉淀业务对象、流程、菜单、看板、系统配置。
3. 租户克隆：从模板租户快速生成新租户，减少重复建设。
4. 扩展模块：通过 extensions/ 目录 + Maven Profile 实现客户代码物理隔离，包名约定。
5. 场景脚本目录：把行业场景固化为可复用脚本。
6. 资产沉淀机制：把每一次交付经验转化为可调用资产，反哺到模板租户、场景脚本、组件库。

### 与 FDE 工作闭环的对应

1. 第一步进入现场：业务对象和关系在租户里已建模，FDE 现场理解可以快速落到结构化输入。
2. 第二步需求解构：FDE 直接调用 MCP 工具链，按业务对象清单和数据关系矩阵解构客户语言。
3. 第三步方案架构：判断低代码、Web2、插件、Agent、外部系统对接，依赖的是分层清晰的平台能力。
4. 第四步快速搭建：调用 MCP 工具链创建表单、发布菜单、授权角色、发布流程，骨架快速跑通。
5. 第五步现场迭代：调用 MCP 工具链修正字段、调整流程、改动菜单权限。
6. 第六步业务验证：调用 MCP 工具链执行端到端测试，验证业务闭环。
7. 第七步资产沉淀：把这一次的工具、模板、知识反哺回场景脚本目录、模板租户、长期记忆。
