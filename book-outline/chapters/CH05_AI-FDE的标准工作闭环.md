# CH05 AI-FDE 的标准工作闭环

## 章节定位

把 FDE 从定义落到工作方法，说明一个真正的 AI-FDE 如何从现场问题推进到可运行、可验证的交付成果。

## 核心问题

一个真正的 AI-FDE 如何完成项目？

## 核心观点

1. FDE 的工作不是写代码，也不是简单做出系统，而是把业务问题推进为可运行、可验证的交付成果。
2. 标准闭环包括现场理解、业务建模、方案架构、基础设施调用、上线验证、持续迭代、成果确认和资产沉淀。
3. 每一步都必须同时考虑业务价值、技术可行性、交付效率和复用价值。

## 小节结构

1. 进入现场：识别真实业务场景和关键角色。
2. 需求解构：把客户语言转成业务对象、数据关系和流程节点。
3. 方案架构：判断数据治理、Web2、后端插件、AI Agent 和外部系统对接组合。
4. 快速搭建：数据结构、流程、菜单、权限、联动、接口、场景页面。
5. 现场迭代：用真实反馈修正系统。
6. 业务验证：验证端到端业务闭环。
7. 资产沉淀和基础设施反哺。

## 事实底稿

### 本章事实判断

1. 第四章已经把 FDE 定义为对“需求到交付成果”完整负责的角色。本章把这一抽象定义落到具体工作步骤。
2. 标准工作闭环不是流程图，而是真实工作里每天都在做的几件事：进入现场、需求解构、方案架构、快速搭建、现场迭代、业务验证、资产沉淀、基础设施反哺。
3. 每一步都同时承担两种职责：把这一次项目做对，把这一次项目沉淀成下一次可调用的资产。
4. AI 在闭环里的位置不是某一两个步骤的工具，而是贯穿所有步骤的认知和执行底座。
5. 闭环的最终判定不是“系统能跑”，而是“客户业务是否真的发生了变化”：用户是否用起来、关键指标是否改善、运维是否可承接、后续调整是否可行。
6. 闭环不是串行一次性的，而是现场反馈驱动的多次迭代：数据先跑通、流程先穿透、用户先试用、模型先接触真实输入，异常先暴露，再不断修正。
7. 闭环成立的前提是 FDE 同时具备三件事：现场理解能力、基础设施调用能力和工程实施能力。任何一项缺位，闭环都会断。

### 原始大纲可用观点

1. 工作形态：单人全权闭环、无岗位交接、全流程穿透。
2. 标准全链路工作内容：业务洞察、方案架构、AI 搭建、系统集成、上线迭代、业务运营、资产沉淀、产品反哺。
3. 相比传统模式的碾压式优势：零信息损耗、极低沟通成本、极致交付迭代效率。
4. AI 不是辅助，是 FDE 单人闭环落地的核心生产力底座。

### 团队实践可用素材

1. 团队资源数字化平台、MCP 工具链、Web2 框架作为“三层调用面”，是 FDE 在现场真正动用的工具。
2. `system-build-workflow-guide.md` 已经把交付拆为 13 个串行阶段：租户与场景确认、需求文档解析、业务对象建模、数据关系矩阵、表单字段设计、依赖排序与创建、关联表单与跟随联动补配、表单审计与修复、菜单发布与图标校验、角色菜单授权、BPM 流程设计与发布、端到端业务测试、输出 Web2 下一阶段输入材料。
3. 这些阶段不是 FDE 闭门造车的脚本，而是把现场业务问题转换为可运行系统的工程化路径。
4. 第三章运维材料中“流程定义文件缺失但数据库记录仍在”“业务编码和代码比较值不一致”“联动配置中字段名、条件规则、任务表不匹配”等案例，本质上都说明：业务、数据、流程、配置、代码和运行环境之间的关系，必须从一开始就建对，而不是事后修补。
5. 业务对象建模要求区分主数据、业务单据、明细数据、显示字段四类，避免重复维护和孤立数据。
6. 数据关系矩阵要求在创建表单前先明确所有表单之间的引用关系和创建拓扑，避免循环引用和后置补配过多。
7. 端到端业务测试不能只看 API 创建成功，必须覆盖表单 CRUD、关联表单弹窗、跟随联动、菜单可见、图标显示、流程发起、审批、结束和数据清理。
8. CRM 场景下，FDE 在使用 MCP 工具链时，串行执行审计与修复、菜单权限审计、流程冒烟测试三类脚本，可以一次性完成一类系统的端到端验收。

### 闭环七步的内部关系

1. 进入现场：把客户语言中的矛盾、犹豫、优先级、表面诉求一起带回来。
2. 需求解构：把客户语言转成结构化的业务对象、数据关系、流程节点、角色权限、报表与定制交互候选。
3. 方案架构：判断低代码、Web2、后端插件、AI Agent、外部系统对接各承担多少。
4. 快速搭建：调用基础设施创建数据结构、流程、菜单、权限、联动、接口和场景页面，先把骨架跑通。
5. 现场迭代：用真实用户和真实输入修正系统，让异常先暴露在过程中而不是项目末期。
6. 业务验证：验证端到端业务闭环是否真的改变业务，而不是停留在功能可用。
7. 资产沉淀和反哺：把这一次的模板、组件、脚本、知识、排障经验沉淀回基础设施，让下一个项目更快起步。

### 每一步的双重职责

1. 现场理解：识别真实问题（为这一次）+ 抽象出可复用的行业判断（为下一次）。
2. 需求解构：完成当前项目结构（为这一次）+ 形成业务对象和关系模板（为下一次）。
3. 方案架构：选择可行的技术组合（为这一次）+ 沉淀架构决策库（为下一次）。
4. 快速搭建：交付可运行系统（为这一次）+ 抽取通用工具脚本和场景脚本（为下一次）。
5. 现场迭代：修正当前系统（为这一次）+ 形成可被 Agent 调用的排障脚本（为下一次）。
6. 业务验证：证明成果可验证（为这一次）+ 形成可复用的验证用例和指标体系（为下一次）。
7. 资产沉淀和反哺：把项目经验转化为可调用资产，把资产回流到工具链、数据治理、组件库、长期记忆。

### 与第三章、第四章的对照

1. 第三章的传统交付“五种损耗”在本章的七步里要反向消解：信息损耗在进入现场消解，沟通损耗在方案架构消解，迭代损耗在现场迭代消解，资产损耗在资产沉淀和反哺消解，运维损耗在业务验证和持续运营接口消解。
2. 第四章的“六条不做边界”在七步里要逐条遵守：进入现场不替代客户决策、需求解构不替代领域专家、方案架构不替代基础设施、快速搭建不替代标准研发、现场迭代不替代风险审查、业务验证不替代组织变革、资产沉淀不替代治理。
3. 第四章的“组织必须给 FDE 三件事：授权、基础设施、资产沉淀机制”在本章七步里要全部落到动作：授权体现在 FDE 可以自己决定方案方向，基础设施体现在 FDE 直接调用平台和工具链，资产沉淀机制体现在 FDE 每一步都同时沉淀资产。

### 需要脱敏或加工的内容

2. CRM 场景只作为“可被快速搭建的示例”，不写成案例履历。
3. 风洞排程不写具体子系统数量和算法细节，留给第十一章案例章展开。

### 本章论证链

1. 上承 CH04：FDE 是责任链条重构，本章给出重构如何具体运作。
2. 第一步进入现场：识别真实业务场景和关键角色，承接第三章“信息损耗下降”。
3. 第二步需求解构：把客户语言转成业务对象、数据关系和流程节点，承接第二章“一个人承担过去一支小队工作”的具体形态。
4. 第三步方案架构：判断低代码、Web2、后端插件、AI Agent、外部系统对接的组合，给出技术分工的判断框架。
5. 第四步快速搭建：数据结构、流程、菜单、权限、联动、接口、场景页面，体现基础设施调用能力。
6. 第五步现场迭代：用真实反馈修正系统，体现 AI 时代的迭代前置原则。
7. 第六步业务验证：验证端到端业务闭环，体现“系统只是手段，成果才是终点”的成果优先原则。
8. 第七步资产沉淀和基础设施反哺：把项目经验转化为组织资产，承接第四章深层价值。
9. 收束在一个明确判断：闭环不是线性流程，而是现场反馈驱动的多次迭代；它的最终判定不是“系统能跑”，而是“客户业务是否真的发生了变化”。
10. 过渡到 CH06：这种闭环要求一种全新的人才能力模型，下一章拆解它。
