# CH10 基础设施驱动的交付工作流：从需求到交付成果

## 章节定位

下篇第二章。承接 CH09 基础设施，进入"基础设施如何变成实际交付流程"。覆盖 13 步工作流骨架，但保持为 FDE 工作方法叙述。

## 核心问题

一个产业需求如何被推进为可运行、可验证的交付成果？

## 核心观点

1. 基础设施驱动的交付不是从页面开始，也不以系统上线为终点，而是从业务对象、数据关系和流程事实开始，最终走向用户成果。
2. 数据治理阶段负责稳定数据结构、流程、菜单、权限和运行态验证。
3. Web2 定制前端是在数据结构和业务闭环稳定后的增强层；系统只是载体，成果才是终点。
4. 13 步工作流不是死顺序，而是反馈驱动的执行骨架；每一步都要回头看上一一步是否稳定。
5. 工作流的真正落点不是"系统通过测试"，而是"业务结果可被验证、用户可持续使用"。

## 小节结构

1. 从需求到交付成果：工作流要回答的真问题。
2. 租户与场景确认：明确本次构建在哪个上下文里跑。
3. 需求文档解析：把客户语言翻译为结构化输入。
4. 业务对象建模：识别主数据、业务单据、明细数据、展示字段。
5. 数据关系矩阵设计：明确表单之间的引用关系和创建拓扑。
6. 表单字段详细设计：业务含义、数据来源、组件类型、是否参与流程。
7. 依赖排序与表单创建：按拓扑顺序串行创建表单。
8. 关联关系与联动补配：补齐创建阶段漏写的跟随联动和审计修复。
9. 菜单、权限、流程发布：让数据结构被角色真正访问。
10. 端到端业务测试：用真实操作链路验证系统可用。
11. 交付成果确认：业务闭环是否跑通、用户是否可持续使用、关键结果是否可验证。
12. 输出 Web2 下一阶段输入材料：低代码交付不等于 Web2 启动。
13. 工作流是反馈驱动的：每一步都要回头看。

## 事实底稿

### 本章事实判断

1. CH10 是下篇第二章，从基础设施转到工作流。基础设施是底盘，工作流是底盘上的实际操作；底盘不稳，工作流跑不动；底盘再好，没有工作流也出不了成果。
2. 工作流的起点不是"页面"，而是"业务对象、数据关系、流程事实"。这三个东西决定后续所有动作的稳定性和成本。
3. 工作流的终点不是"系统通过测试"，而是"业务闭环跑通、用户持续使用、关键结果可验证"。这是成果优先原则在工作流层面的具体落点。
4. 13 步工作流（租户与场景确认 → 需求文档解析 → 业务对象建模 → 数据关系矩阵设计 → 表单字段详细设计 → 依赖排序与表单创建 → 关联关系与联动补配 → 表单审计与修复 → 菜单发布与图标校验 → 角色菜单授权 → BPM 流程设计与发布 → 端到端业务测试 → 输出 Web2 下一阶段输入材料）是 FDE 在低代码阶段的固定骨架。
5. 这 13 步不是"每一步做完再做下一步"的串行死顺序，而是反馈驱动的执行链。每一步都要回头看上一步的产物是否稳定，不稳定就先回头，不要硬推进。
6. 租户与场景确认是工作流的第零步：明确本次构建在哪个租户、哪个业务场景、系统边界在哪里、交付范围是什么。FDE 进入现场之后第一件事不是写代码，而是确认这些元数据。
7. 需求文档解析把客户语言翻译成结构化输入：业务模块、业务对象、业务动作、角色、流程、查询报表、定制交互候选、接口缺口初判。这一步决定后面所有动作能不能跑通。
8. 业务对象建模判断哪些对象独立建表、哪些只是引用或展示：主数据、业务单据、明细数据、展示字段。这四类一旦分错，整个数据结构就崩。
9. 数据关系矩阵是工作流的关键工程产物：来源表单、字段 ID、目标表单、关系、显示字段、带出字段、是否只读。它能直接推导出表单的创建拓扑。
10. 表单字段详细设计是字段级别的工程化：每个字段要回答业务含义、数据来源、数据类型、是否枚举、是否引用业务对象、是否只是展示、是否参与流程或统计。
11. 依赖排序与表单创建按"先主数据、再被引用对象、再业务单据、最后后置表单"的拓扑顺序串行创建，并尽量一次性写入已知关联配置。
12. 关联关系与跟随联动补配补齐创建阶段漏写的关联字段和跟随联动。`leaderFormId` 必须是本表单字段 ID，不是目标表单 ID——这是审计和修复阶段最容易踩的坑。
13. 表单审计与修复在每建完一步后立即检查实际平台保存结果。失败示例：组件类型丢失、leaderFormId 错误、关联字段不完整。
14. 菜单、权限、流程发布让数据结构被角色真正访问：业务菜单只允许两级，路径分别使用 lowcode/dynamicInfo/{formId} 和 lowcode/dynamicWorkflow/{modelId}，菜单图标使用平台标准格式；流程模型由开始、用户任务、网关、结束组成，节点分配规则匹配用户任务数量。
15. 端到端业务测试用真实操作链路验证系统可用：表单 CRUD、关联表单选择、跟随联动、菜单可见、图标显示、流程发起、审批、结束、数据清理。不能只看 API 创建成功，必须验证前端真实依赖字段。
16. 交付成果确认是工作流的真正落点：业务闭环是否跑通、用户是否可持续使用、关键结果是否可验证。系统只是载体，成果才是终点。
17. 输出 Web2 下一阶段输入材料是低代码阶段的收口动作：表单与数据结构清单、关联关系清单、流程清单、菜单和权限清单、可复用接口清单、Web2 候选页面、后端接口缺口初判。
18. 工作流是反馈驱动的：每一步都要回头看。这一判断来自工程实践——前一步没稳定就硬推进，后端阶段必然大返工。
19. 这套工作流在团队里是标准化的：FDE 在现场不必每一步都从零写代码，而是调用 MCP 工具链完成。但工具链是手段，13 步骨架才是 FDE 工作方法的真正形式。
20. 工作流和 FDE 工作闭环的关系：第五章讲的是 FDE 与客户/业务/团队的关系侧（七步业务闭环），第十章讲的是 FDE 与平台/工具/数据结构的关系侧（13 步技术骨架）。两者合起来，才是完整的 FDE 工作方法。

### 原始大纲可用观点

1. 团队资源数字化平台 + MCP 工具链 + 13 步工作流 = FDE 在低代码阶段的固定执行骨架。
2. 工作流的起点是业务对象、数据关系、流程事实，而不是页面。
3. 工作流的终点是用户成果可被验证，而不是系统上线。
4. 13 步不是死顺序，而是反馈驱动的执行链。
5. 数据结构治理的稳定，决定工作流上层所有动作的稳定。
6. 端到端测试不是看 API 创建成功，而是看前端真实依赖字段能不能跑通。
7. 低代码阶段只输出 Web2 下一阶段输入材料，不直接做 Web2 页面。
8. 后端接口缺口只记录不实现，避免增加维护成本和权限风险。
9. 工作流每一步都要回头看上一步产物是否稳定，不要硬推进。
10. 工作流和 FDE 七步业务闭环对应，前者是技术骨架，后者是业务骨架。

### 团队实践可用素材

1. 13 步工作流作为骨架：租户与场景确认 → 需求文档解析 → 业务对象建模 → 数据关系矩阵设计 → 表单字段详细设计 → 依赖排序与表单创建 → 关联关系与跟随联动补配 → 表单审计与修复 → 菜单发布与图标校验 → 角色菜单授权 → BPM 流程设计与发布 → 端到端业务测试 → 输出 Web2 下一阶段输入材料。
2. 租户与场景确认包括：租户 ID、登录账号、系统名称、系统边界（当前阶段只做低代码，Web2 下一阶段）、交付范围（表单、菜单、权限、流程、数据联动、测试数据、Web2 输入材料）。
3. 需求文档解析的八个产物：业务对象清单、表单候选清单、流程候选清单、菜单模块初稿、角色权限初稿、Web2 下一阶段输入材料初稿、后端接口缺口初判。
4. 业务对象建模的四类：主数据（长期存在被多业务复用）、业务单据（一次业务动作或过程记录）、明细数据（从属单据的多行数据）、展示字段（关联对象的冗余信息）。
5. 数据关系矩阵的模板：来源表单、字段 ID、目标表单、关系、显示字段 dlabel、带出字段、是否只读。关键规则：associationForm 只表达关联到哪个目标表单，followChange 表达跟随本表单哪个字段变化。
6. 表单字段详细设计的七个问题：业务含义、数据来源、数据类型、是否固定枚举、是否引用业务对象、是否只是展示、是否参与流程或统计。
7. 组件选择原则：短文本用 input，长文本用 textarea，日期用 date，数量金额用 number，固定枚举用 select/radio，布尔用 switch/radio，用户用 selUser，业务对象引用用 associationForm，明细列表用 dynamicTable。
8. 依赖排序的推荐顺序：无依赖主数据 → 被依赖基础对象 → 业务单据 → 后置表单。
9. 关联字段配置：associationForm 写入 tableId/formId/dval/dlabel/title。
10. 跟随字段配置：followChange 写入 leaderFormId（本表单字段 ID）、dlabel、readonly。
11. 表单审计检查项：表单数量、字段数量、组件类型、枚举选项、关联配置、跟随联动、只读展示字段。
12. 常见失败示例：leaderFormId 错误地写成了目标表单 ID；select/date/number 保存成 input；菜单图标空白；流程无法运行（无审批人或任务不流转）。
13. 菜单层级规则：业务菜单只允许两级，模块目录挂在平台根目录下（parentId=0）；表单菜单路径 lowcode/dynamicInfo/{formId}，流程菜单路径 lowcode/dynamicWorkflow/{modelId}；图标使用 png-* 平台标准格式。
14. 角色菜单授权常见问题：只授权子菜单未授权父菜单；表单菜单授权了但流程菜单漏了。
15. BPM 流程构建：流程 key/name、绑定表单（formType 和 formId）、BPMN XML（开始、用户任务、结束、网关）、用户任务（节点 ID、名称、formKey）、分配规则、发布版本、流程菜单。
16. 端到端测试范围：表单 CRUD、关联表单选择、跟随联动、菜单可见、图标显示、流程发起、流程审批、流程结束、数据清理。
17. 交付检查清单：18 项交付前必须检查的事项，包括需求文档解析、数据关系矩阵、字段设计、表单创建、关联配置、跟随字段、菜单发布、图标审计、角色授权、流程发布、Web2 输入材料、可复用接口、CRUD 测试、关联测试、流程测试、数据清理、文档更新等。
18. 常见风险：字段类型丢失、关联字段不完整、跟随联动配置错误、菜单不可见、图标不显示、流程无法运行、并发更新表单、低代码阶段过早进入 Web2、Web2 输入材料不足、随意新增后端接口。
19. 系统构建配置化方向：每个业务系统建议沉淀为独立场景目录，含 system.config.json、build.workflow.json、build-state.json、requirements/、forms/、menus/、workflows/、permissions/、artifacts/、tests/、docs/ 等子目录。
20. 固定工作流 CLI：plan/audit/run/resume/status 等命令；scaffold 创建标准目录，validate 离线校验，audit 只执行带 audit 标签的阶段，--dry-run 只打印计划不写状态文件。

### 13 步工作流的内部关系

1. 第零步租户与场景确认是上下文对齐，决定整个工作流跑在哪。
2. 第一到第三步（需求文档解析 → 业务对象建模 → 数据关系矩阵设计）是结构化设计阶段，决定骨架。
3. 第四到第七步（表单字段详细设计 → 依赖排序与表单创建 → 关联关系与联动补配 → 表单审计与修复）是工程化执行阶段，决定骨架落到平台。
4. 第八到第十步（菜单发布与图标校验 → 角色菜单授权 → BPM 流程设计与发布）是企业级封装阶段，让骨架被角色真正访问。
5. 第十一步端到端业务测试是运行态验证，让骨架被业务实际跑通。
6. 第十二步输出 Web2 输入材料是低代码收口，把骨架作为下一阶段的输入。
7. 第十三步工作流是反馈驱动的判断，是工程化执行阶段和企业级封装阶段的反复循环。
8. 这五段不是平行的清单，而是嵌套的依赖：上下文错了所有动作错；骨架错了工程返工；工程错了封装错；封装错了业务跑不动；业务跑不动，Web2 阶段也做不出成果。

### 工作流和七步业务闭环的对应

1. 七步闭环第一步进入现场 = 13 步工作流第零步租户与场景确认。
2. 七步闭环第二步需求解构 = 13 步工作流第一到第三步（需求文档解析、业务对象建模、数据关系矩阵设计）。
3. 七步闭环第三步方案架构 = 13 步工作流覆盖范围的整体判断（哪些进低代码、哪些进 Web2、哪些进定制）。
4. 七步闭环第四步快速搭建 = 13 步工作流第四到第十步（表单创建、关联补配、菜单权限流程）。
5. 七步闭环第五步现场迭代 = 13 步工作流第七、八步（表单审计、菜单权限修复）。
6. 七步闭环第六步业务验证 = 13 步工作流第十一步端到端业务测试和第十二步交付成果确认。
7. 七步闭环第七步资产沉淀 = 13 步工作流第十二步输出 Web2 输入材料和第十三步反馈驱动判断的反哺。
8. 七步业务闭环是"和客户/业务/团队一起走"的叙述；13 步技术骨架是"和平台/工具/数据结构一起干"的执行。两者合起来，才是完整的 FDE 工作方法。

### 反馈驱动的具体内涵

1. 表单创建阶段回头看数据关系矩阵是否完整；矩阵漏了某个对象，先补矩阵再继续。
2. 关联补配阶段回头看表单创建阶段的关联字段是否正确写入；写入错误的，先修复再补配。
3. 表单审计阶段回头看字段详细设计是否清晰；不清晰的，先回到字段设计再审计。
4. 菜单发布阶段回头看表单创建是否完成；没创建完的，先回到表单创建再发布。
5. 角色授权阶段回头看菜单发布是否成功；菜单发布失败的，先回到菜单发布再授权。
6. 流程发布阶段回头看角色菜单授权是否完成；授权漏的，先回到授权再发布流程。
7. 端到端测试阶段回头看业务对象建模是否分错类；分错的，先回到建模再测试。
8. 反馈驱动的核心是"不要硬推进"，硬推进的代价是后端阶段大返工。
9. 团队实践里常见的反馈驱动错误：跳过表单审计直接进入菜单发布；跳过端到端测试直接上线；跳过交付成果确认直接进入 Web2。

### 交付成果确认的具体内涵

1. 业务闭环是否跑通：用户从菜单进入，能完成 CRUD，能走完流程，能看到结果。
2. 用户是否可持续使用：用户第二天、第三天还能正常使用，没有报错、没有权限阻挡、没有数据残留。
3. 关键结果是否可验证：业务的关键指标是否被记录、是否可被查询、是否可被复盘。
4. 系统只是载体：系统跑通不等于业务跑通，业务跑通不等于用户真正采用。
5. 成果优先原则的落点：FDE 在交付阶段问的不是"系统上线了吗"，而是"用户用起来了吗、结果变化了吗"。

### 输出 Web2 输入材料的具体内涵

1. 表单与数据结构清单：formId、tableId、字段、组件类型。
2. 关联关系清单：associationForm 的目标表单、显示字段和值字段。
3. 流程清单：modelId、processKey、绑定表单、用户任务。
4. 菜单和权限清单：菜单 ID、path、授权角色。
5. 可复用接口清单：低代码动态表单接口和 BPM 接口。
6. Web2 候选页面：需求中低代码前端无法满足的交互点。
7. 后端接口缺口初判：只做判断，不在本阶段新增接口。
8. 这一步是低代码阶段的收口动作，不是 Web2 阶段的启动动作。

### 13 步工作流的工程化价值

1. 它把 FDE 在低代码阶段的工作固化下来，新人按这套骨架跑就能达到工作基线。
2. 它让 FDE 在不同项目之间复用同一套执行步骤，避免每个项目都重新发明工作流。
3. 它让团队能积累跨项目的工程资产：场景脚本、配置模板、审计脚本、测试用例。
4. 它让基础设施的能力真正被 FDE 调度起来，而不是停留在演示阶段。
5. 它把"AI 时代的软件更容易做出"的命题落到具体执行步骤里——容易做出的是骨架，真正难的是让用户用起来。

### 工作流的反馈驱动的工程化体现

1. 每个阶段都有 audit 标签，可以单独执行审计而不直接执行修复。
2. --dry-run 只打印计划不写状态文件，让 FDE 在执行前先看清楚。
3. --from 可以从某一步开始执行，便于回退修复。
4. --stage 只执行某一步，便于单点修复。
5. 同一批表单并发更新会报错，要求串行执行，避免物理表同步竞态。
6. 这些机制在工作流之外提供了"反馈驱动的工程化抓手"，让 FDE 真正能在工程上落地。
