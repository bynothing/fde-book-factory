# CH02 已经发生的变化：一个人开始承担过去一支小队的工作

## 章节定位

继续倒叙，从宏观事实进入交付现场，展示 AI-FDE 带来的具体变化。

## 核心问题

FDE 到底带来了什么肉眼可见的变化？

## 核心观点

1. 过去一个产业软件项目依赖多岗位分段协作。
2. AI Native 交付体系中，一个强 FDE 可以承担业务洞察、方案设计、成果交付、客户推进、上线迭代和资产沉淀。
3. 变化不是因为人突然全能，而是 AI 工具、基础设施和自动化工作流放大了个人能力。
4. 交付效率提升的核心，是信息链条、责任链条和迭代链条被压缩。

## 小节结构

1. 传统交付现场：一件事经过太多人。
2. AI-FDE 现场：一个人对结果负责。
3. 从“写代码更快”到“责任链条更短”。
4. 团队实践中的变化：多场景、工具链、数字员工。
5. 这种变化为什么不是偶然。

## 事实底稿

### 本章事实判断

1. 第二章不解释 FDE 定义，而是先写交付现场已经出现的变化。
2. 变化的核心不是“一个人什么都会”，而是“一个人对成果负责，并被 AI、基础设施和自动化工具链托举”。
3. 传统产业软件交付依赖销售、售前、产品、架构、研发、测试、实施、运维分段协作；AI-FDE 体系把其中大量转述、等待、交接和重复执行压缩到一个责任闭环中。
4. AI 对 FDE 的放大，不只体现在代码生成，而体现在调研、方案、架构、开发、调试、文档、运维、复盘、资产沉淀等全链路。
5. 团队实践中，工具链、团队资源数字化平台、MCP、Web2 定制前端、数字员工运维共同形成了“单人闭环”的环境。
6. 本章必须继续坚持“从需求到交付成果”，不能写成“一个人更快做系统”。

### 原始大纲可用观点

1. 真 FDE 依托 AI 工具链与自研交付底座，单人完成业务洞察、方案架构、AI 搭建、系统集成、上线迭代、业务运营、资产沉淀和产品反哺。
2. FDE 核心价值是降低信息损耗、沟通成本和交付迭代成本。
3. 今天的 FDE 不是靠人的天赋变全能，而是靠 AI 把一个人武装成一支团队。
4. FDE 不再做大量重复执行工作，而是把精力集中在决策、架构、判断、价值提炼和客户对齐。

### 团队实践素材

1. `yunjing-mcp` 的工程结构显示，工具链被拆成平台通用能力和业务场景脚本：
   - `platform/` 放通用能力，包括 API client、表单、菜单、图标、流程、权限、用户、角色等服务。
   - `scenarios/` 放业务场景脚本，包括 AI CRM、CRM 535、风洞排程、英语学习、高考词汇、酒店等。
   - `docs/` 放系统构建、BPM、表单、菜单、联动、Web2 交接等方法论和模板。
2. `MCP_INDEX.md` 显示，工具链已经覆盖：
   - BPM Admin CLI。
   - 系统构建 CLI。
   - 表单数据 CRUD CLI。
   - 表单批量创建。
   - 菜单批量创建。
   - 角色菜单授权。
   - 菜单图标审计修复。
   - 微信公众号发布 CLI。
3. 这些工具链说明：FDE 不是从空白处写系统，而是在调用已有基础设施、脚本、模板和自动化能力。
4. 运维实践中，数字员工已承担多环境巡检、数据库备份检查、证书管理、CI/CD、OSS、云资源、故障排查、构建部署和群通知等职责。
5. 运维月度材料显示，在 2026-05-08 到 2026-06-09 约 33 天内，数字员工运维处理了多类故障、构建部署和流程优化：
   - 修复后端 7+ 起重大故障。
   - 提交 3 个后端源码 PR。
   - 部署 13+ 次 Web2 子系统。
   - 数据库备份 30+ 次自动化。
   - 巡检报告 30+ 份。
   - 处理用户咨询 100+ 条。
   - 飞书群消息 200+ 条，形成可追溯记录。
6. 这些数据可作为“过去需要多人协作的持续运营能力，正在被数字员工和工具链压缩”的证据。

### 需要脱敏或加工的内容

1. 不写服务器 IP、账号、端口、真实访问地址。
2. 不直接展开生产环境名称清单。
3. 不写内部凭证、令牌、登录方式。

   - 某道面检测项目。
   - 某维修追溯项目。
   - 某复杂实验室排程项目。
   - 某酒店投资管理场景。
5. 运维案例不写成流水账，抽象成“诊断、决策、执行、验证、收尾”的闭环。

### 本章论证链

1. 第一章已经说明产业 AI 进入落地分水岭，问题从模型转向成果。
2. 第二章落到现场：成果交付要求责任链条变短。
3. 传统方式中，需求要经过多人转述，信息损耗大，迭代慢。
4. AI-FDE 体系中，一个人不是替代所有岗位，而是把原来割裂的责任重新合并。
5. 合并责任的前提，是背后有基础设施、工具链、脚本、模板、数字员工和组织机制托举。
6. 因此，本章的结论是：变化不是“一个人更强”，而是“一个人被一整套 AI Native 生产力体系武装后，可以对成果形成闭环责任”。
